# M0: 邀请码注册机制

> 优先级：P0（最高，所有模块的前置）
> 定位：内部团队工具，禁止公开注册，必须通过管理员邀请码注册

## 需求描述

- 关闭公开注册，注册页必须填写有效邀请码
- 管理员可在管理页面生成、查看、停用邀请码
- 邀请码可设置：有效期、最大使用次数、备注
- 注册成功后邀请码使用次数 +1，达到上限自动失效

## 数据库设计

### 新建 invitation_codes 表

```sql
CREATE TABLE IF NOT EXISTS invitation_codes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  code TEXT NOT NULL UNIQUE,                    -- 邀请码（8位随机字母数字）
  created_by INTEGER REFERENCES users(id) ON DELETE SET NULL,
  note TEXT DEFAULT NULL,                       -- 备注（发给谁的）
  max_uses INTEGER DEFAULT 1,                   -- 最大使用次数（1=一次性，0=无限）
  used_count INTEGER DEFAULT 0,                 -- 已使用次数
  expires_at DATETIME DEFAULT NULL,             -- 过期时间（NULL=永不过期）
  is_active INTEGER DEFAULT 1,                  -- 是否启用（管理员可手动停用）
  created_at DATETIME DEFAULT (datetime('now')),
  updated_at DATETIME DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_invitation_codes_code ON invitation_codes(code);
```

### 新建 invitation_usage 表（记录使用明细）

```sql
CREATE TABLE IF NOT EXISTS invitation_usage (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  code_id INTEGER NOT NULL REFERENCES invitation_codes(id),
  user_id INTEGER NOT NULL REFERENCES users(id),  -- 用此邀请码注册的用户
  used_at DATETIME DEFAULT (datetime('now'))
);
```

## 邀请码生成规则

- 格式：8 位大写字母+数字（`/^[A-Z0-9]{8}$/`），如 `SD2X7KM9`
- 前缀可选：`SD-` 开头方便识别，如 `SD-X7KM9A`
- 字符集去除易混淆字符（0/O、1/I），实际使用 `ABCDEFGHJKLMNPQRSTUVWXYZ23456789`（30 个字符）
- 生成逻辑：

```javascript
const CHARSET = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789';
function generateInvitationCode() {
  const bytes = crypto.randomBytes(8);
  return Array.from(bytes).map(b => CHARSET[b % CHARSET.length]).join('');
}
```

> **注意**：原方案使用 `crypto.randomBytes(5).toString('base64url').substring(0,8).toUpperCase()`，但 base64url 编码会产生 `-` 和 `_` 字符，不符合 `[A-Z0-9]` 格式要求，已修正为显式字符集映射。

## API 设计

### 邀请码管理（仅管理员）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/admin/invitation-codes` | 获取所有邀请码列表 |
| POST | `/api/admin/invitation-codes` | 生成新邀请码 |
| PUT | `/api/admin/invitation-codes/:id` | 更新邀请码（停用/修改备注） |
| DELETE | `/api/admin/invitation-codes/:id` | 删除邀请码 |

**POST 请求体：**
```json
{
  "note": "给张三",
  "max_uses": 1,
  "expires_at": "2026-05-01T00:00:00Z"   // 可选
}
```

**GET 响应：**
```json
{
  "codes": [
    {
      "id": 1,
      "code": "SD-X7KM9A",
      "note": "给张三",
      "max_uses": 1,
      "used_count": 0,
      "expires_at": null,
      "is_active": true,
      "created_at": "2026-04-16T10:00:00Z",
      "usedBy": []
    }
  ]
}
```

### 注册接口改造

**POST /api/auth/register（修改）**

请求体增加 `invitation_code` 字段：
```json
{
  "email": "user@example.com",
  "password": "xxx",
  "invitation_code": "SD-X7KM9A"
}
```

校验逻辑（**整个流程必须在 `db.transaction()` 中执行**，保证原子性）：
1. 查询邀请码是否存在且 `is_active = 1`
2. 检查 `used_count < max_uses`（`max_uses = 0` 表示无限）
3. 检查 `expires_at` 是否过期（NULL 表示永不过期）
4. 校验通过 → 注册用户 → `used_count += 1` → 记录 `invitation_usage`（均在同一事务内）
5. 校验失败 → 返回 `400 { error: "邀请码无效或已过期" }`

> **事务保证**：步骤 4 中的"创建用户 + 更新 used_count + 写入 invitation_usage"必须包装在 better-sqlite3 的 `db.transaction()` 中，避免部分成功导致数据不一致。

## 验收标准

- [ ] 无邀请码无法注册（返回错误提示）
- [ ] 有效邀请码可正常注册
- [ ] 过期邀请码无法使用
- [ ] 已用完的邀请码无法使用
- [ ] 管理员可生成、查看、停用、删除邀请码
- [ ] 邀请码使用记录可追溯（谁用了哪个码）
- [ ] 现有已注册用户不受影响
