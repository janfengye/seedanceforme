## 技术约束

- 后端：纯 JavaScript ESM（非 TypeScript）
- 数据库：SQLite（better-sqlite3），需写迁移脚本
- 前端：React 19 + TypeScript + Tailwind CSS
- 部署：Docker 自定义构建
- 无测试框架、无 linter

## 调研结论

> 基于对 Kitsu、ShotGrid、AYON、Prism Pipeline、Runway ML、Pika Labs、Kling AI、Figma 等工具的深度调研。
> 完整调研报告见 [调研-社区项目深度分析.md](./调研-社区项目深度分析.md)。

### 核心发现

1. **层级结构验证通过**：行业标准为 Project → Sequence → Shot → Version（Kitsu/ShotGrid 均如此）。我们的 Project → Episode → Shot → Version 三级结构是合理的简化——省略了 Sequence 层，因为 AI 视频生成场景中镜头通常是独立创意单元，不需要场景级分组。

2. **权限模型验证通过**：Kitsu 有 6 级角色（Studio Manager 到 Client），ShotGrid 有 4+ 级角色，Figma 有 4 级。行业 RBAC 最佳实践建议"初期从 2-3 个角色开始"。我们的 admin/user 二元模型完全适合当前阶段（<50 用户），且已通过 `team_members.role` 预留了团队级角色扩展路径。

3. **文件命名验证通过**：Netflix VFX 规范使用 `{showID}_{episode}_{shot}_{task}_{vendor}_{version}` 格式。我们的 `{项目代号}-{集数}-{镜头号}-{版本号}-{用户昵称}.mp4` 与之异曲同工，去除了不适用于 AI 视频的字段（帧范围、任务类型），字母版本号（A-Z）比数字版本号（v001）更适合轻量场景。

4. **draft/output 模型是独有创新**：传统工具的 Version 是 1:1 的（一次 publish 对应一个 version）。我们的"一个 draft 展开多个 output"模式是 AI 视频生成特有的需求，在行业中没有直接对应，设计合理。

5. **AI 视频平台团队功能普遍不成熟**：Runway ML 有 Enterprise workspace 但价格高昂，Pika 依托 Discord 社区，Kling 仅有基础多用户。Seedance 2.0 的团队协作升级在同类工具中属于差异化优势。

### 方案信心评估

| 模块 | 信心度 | 说明 |
|------|--------|------|
| M1 用户昵称 | 高 | 所有行业工具都有用户标识，昵称是基础需求 |
| M2 即梦账号隔离 | 高 | 类似 SaaS 的 API Key 管理，admin 专属 + user 摘要是标准做法 |
| M3 项目结构改造 | 高 | 三级结构经行业验证，省略 Sequence 合理 |
| M4 版本号与文件命名 | 高 | 命名格式与 Netflix/ShotGrid 规范一致，字母版本号是合理简化 |
| M5 管理员统计面板 | 中 | 需求明确，但优先级可灵活调整 |
| M6 团队机制 | 中 | 预留设计合理，但实际需求需观察用户增长 |
