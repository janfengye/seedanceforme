# 账号管理体系方案 — 四角色讨论纪要与最终方案

> 讨论日期：2026-04-16
> 参与角色：产品经理、前端程序员、后端程序员、架构师

---

## 一、讨论纪要

### 1.1 产品经理视角

**核心观点：**

- 当前系统是"个人工具"形态，用户体系扁平（admin / user 二元角色），要升级为团队协作平台，权限模型需要更细粒度但不能过度复杂。
- 用户体验优先级：**先把"昵称 + 即梦账号隔离"做好**（P0），再做团队机制（P2）。团队机制第一阶段只建表、不做 UI，所有人默认同一团队。
- 即梦 SessionID 是核心敏感资产，普通用户绝对不能看到。但普通用户需要知道"系统还有多少可用账号"以判断是否能提交任务。
- 管理员统计面板的价值在于：了解谁在用、用了多少资源、是否有滥用。初期不需要复杂图表，表格即可。
- 注册流程：当前支持邮箱验证码注册，保持不变。新用户注册后自动加入默认团队、角色为 `user`。
- **争议点**：昵称是否全局唯一还是团队内唯一？结论：第一阶段全局唯一（因为只有一个默认团队），第二阶段团队机制上线后改为团队内唯一。

**优先级排序：**
1. P0：用户昵称 + 个人设置页
2. P0：即梦账号权限隔离
3. P1：管理员统计面板
4. P2：团队机制（先建表预留）

### 1.2 前端程序员视角

**核心观点：**

- 当前前端路由：`/login`, `/register`, `/`, `/batch`, `/download`, `/settings`, `/admin`。需新增 `/profile`（个人设置）和 `/admin/dashboard`（统计面板）。
- 全局状态（`AppContext`）已有 `User` 类型但无 `nickname` 字段，需扩展 `src/types/index.ts`。
- `Sidebar` 组件需要增加"个人设置"入口，并在用户名旁显示昵称。
- 设置页（`/settings`）需根据 `user.role` 条件渲染：管理员看完整账号管理，普通用户看摘要。
- **争议点**：个人设置页是否复用 `/settings` 还是独立 `/profile`？结论：独立 `/profile`，`/settings` 保留为系统/即梦设置页。职责分离更清晰。
- 管理面板用简单表格组件，不引入图表库（保持轻量）。

**组件清单：**

| 组件 | 路由 | 说明 |
|------|------|------|
| `ProfilePage.tsx` | `/profile` | 个人设置（昵称、邮箱只读、角色只读） |
| `SettingsPage.tsx`（改造） | `/settings` | 按角色渲染不同内容 |
| `AdminDashboard.tsx` | `/admin/dashboard` | 统计面板（仅管理员） |

**状态管理改动：**
- `AppContext` 无需大改，`User` 类型加 `nickname` 字段即可。
- 昵称更新后需刷新全局 `currentUser`。

### 1.3 后端程序员视角

**核心观点：**

- 当前已有 `authenticate` 和 `requireAdmin` 中间件，权限控制基础设施完备。
- 即梦账号 API（`/api/settings/session-accounts`）当前无角色区分，需按 M2 方案改造：GET 接口按角色返回不同数据，POST/PUT/DELETE 加 `requireAdmin`。
- 用户表新增 `nickname` 字段，迁移脚本需兼容现有数据（`ALTER TABLE ADD COLUMN`）。
- **争议点**：昵称唯一性约束用数据库 UNIQUE INDEX 还是应用层校验？结论：应用层校验（因为 SQLite 的 ALTER TABLE 不支持直接加 UNIQUE 约束，且未来要改为团队内唯一）。数据库层加一个普通索引加速查询。
- 团队表（`teams`, `team_members`）第一阶段就建好，注册流程里自动加入默认团队。
- 管理员统计 API 直接聚合现有 `tasks` 表数据，无需新建统计表。

**安全要点：**
- 即梦 SessionID 是长期凭证，泄露等于账号被盗。API 层面严格过滤，普通用户请求绝不返回 `session_id` 字段。
- 所有 admin API 双重验证：`authenticate` + `requireAdmin`。
- 昵称输入严格校验：`/^[A-Za-z0-9]{2,10}$/`，防注入。

### 1.4 架构师视角

**核心观点：**

- 整体架构不变（Express 单体 + SQLite），当前阶段无需引入复杂的 RBAC 框架。用 `role` 字段二元区分（admin/user）足以覆盖需求。
- 团队机制的设计要为未来多团队做好预留，但不要过早实现。`teams` + `team_members` 建表，`projects` 加 `team_id` 外键，但查询逻辑暂不改动。
- 即梦账号当前是"全局共享"模型（所有团队共用），这在第一阶段合理。未来若需团队级账号隔离，可通过 `jimeng_session_accounts` 加 `team_id` 字段实现。
- **争议点**：是否需要引入"团队角色"（owner/admin/member）？结论：第一阶段不需要，系统级 admin/user 足够。第二阶段团队机制上线时再引入 `team_members.role`。
- 数据库迁移策略：继续用现有的顺序迁移脚本方式（`server/database/migrations/`），注意 `shouldSkipMigration` 兼容逻辑。

**扩展性评估：**
- 当前方案可支撑 50 用户 / 5 团队规模，SQLite 足够。
- 若未来需要支撑更大规模，瓶颈在 Playwright 浏览器实例而非数据库。

## 二、最终方案

### 2.1 角色与权限模型

第一阶段采用简单二元角色：

| 角色 | 标识 | 权限 |
|------|------|------|
| 管理员 | `admin` | 全部功能 + 即梦账号管理 + 统计面板 + 用户管理 |
| 普通成员 | `user` | 个人设置 + 创建/管理自己的项目和任务 + 查看即梦账号摘要 |

权限控制点：

| 资源 | 管理员 | 普通成员 |
|------|--------|---------|
| 即梦账号列表（含 SessionID） | 可见 + 增删改 | 仅摘要（数量/状态） |
| 其他用户数据 | 可查看 | 不可见 |
| 管理面板 `/admin/*` | 可访问 | 403 / 路由重定向 |
| 个人设置 `/profile` | 可访问 | 可访问 |
| 项目/任务 | 全量可见 | 仅自己的 |

### 2.2 数据库 Schema 变更

#### 迁移脚本 1：`xxx_add_nickname.sql`

```sql
-- 用户表新增昵称字段
ALTER TABLE users ADD COLUMN nickname TEXT DEFAULT NULL;

-- 昵称查询索引
CREATE INDEX IF NOT EXISTS idx_users_nickname ON users(nickname);
```

#### 迁移脚本 2：`xxx_add_teams.sql`

```sql
-- 团队表
CREATE TABLE IF NOT EXISTS teams (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  description TEXT DEFAULT NULL,
  created_by INTEGER NOT NULL REFERENCES users(id),
  created_at DATETIME DEFAULT (datetime('now')),
  updated_at DATETIME DEFAULT (datetime('now'))
);

-- 团队成员表
CREATE TABLE IF NOT EXISTS team_members (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  team_id INTEGER NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role TEXT NOT NULL DEFAULT 'member',
  joined_at DATETIME DEFAULT (datetime('now')),
  UNIQUE(team_id, user_id)
);

-- 项目表新增团队关联
ALTER TABLE projects ADD COLUMN team_id INTEGER DEFAULT NULL REFERENCES teams(id);

-- 索引
CREATE INDEX IF NOT EXISTS idx_team_members_team_id ON team_members(team_id);
CREATE INDEX IF NOT EXISTS idx_team_members_user_id ON team_members(user_id);
CREATE INDEX IF NOT EXISTS idx_projects_team_id ON projects(team_id);

-- 创建默认团队（动态获取第一个管理员 ID，避免硬编码）
INSERT INTO teams (name, description, created_by)
VALUES ('默认团队', '系统默认团队', (SELECT id FROM users WHERE role = 'admin' ORDER BY id LIMIT 1));

-- 将所有现有用户加入默认团队
INSERT INTO team_members (team_id, user_id, role)
SELECT 1, id, CASE WHEN role = 'admin' THEN 'owner' ELSE 'member' END
FROM users;

-- 将所有现有项目归属默认团队
UPDATE projects SET team_id = 1 WHERE team_id IS NULL;
```

| 方法 | 路径 | 中间件 | 说明 |
|------|------|--------|------|
| GET | `/api/user/profile` | `authenticate` | 获取当前用户信息（含 nickname） |
| PUT | `/api/user/profile` | `authenticate` | 更新昵称 |

**GET /api/user/profile 响应：**
```json

  "id": 1,
  "username": "zhangsan",
  "email": "zhangsan@example.com",
  "nickname": "ZS",
  "role": "user",
  "created_at": "2026-04-10T10:00:00Z"

```

**PUT /api/user/profile 请求/响应：**
```json
// 请求
{ "nickname": "ZS" }
// 成功响应
{ "success": true, "nickname": "ZS" }
// 失败响应
{ "error": "昵称已被其他用户使用" }
```

- 格式：`/^[A-Za-z0-9]{2,10}$/`

## 三、实施风险与注意事项

### 3.1 数据库迁移风险

- **风险**：`ALTER TABLE` 在 SQLite 中有限制，不支持直接添加 UNIQUE 约束。
- **对策**：昵称唯一性用应用层校验，不依赖数据库约束。
- **风险**：迁移脚本在已有数据库和全新数据库上行为不同（参考已知的 `shouldSkipMigration` bug）。
- **对策**：迁移脚本使用 `IF NOT EXISTS`；新脚本测试两种场景。

### 3.2 向后兼容

- **风险**：`nickname` 字段初始为 NULL，未设置昵称的用户在文件命名等场景需要 fallback。
- **对策**：文件命名 fallback 链：`nickname > email前缀 > userId`。前端在关键操作前检查昵称是否已设置，未设置则提示跳转个人设置页。

### 3.3 即梦账号安全

- **风险**：前端缓存或浏览器开发者工具可能泄露 SessionID。
- **对策**：API 层严格控制，普通用户的响应中不包含任何 SessionID 相关字段。前端不缓存账号数据。

### 3.4 团队机制预留

- **风险**：第一阶段建表但不用，可能导致后续代码忘记维护 `team_members` 的一致性（如注册时忘记加入默认团队）。
- **对策**：在注册服务中加入自动关联逻辑，并在文档中明确标注。

### 3.5 性能

- **风险**：管理员统计 API 在任务量大时可能慢。
- **对策**：利用现有索引（`idx_tasks_status`、`idx_tasks_project_id`），查询加时间范围限制。SQLite 在万级记录下无压力。

## 四、社区调研补充

> 基于对 Kitsu、ShotGrid、Figma 等工具的深度调研，验证并补充本方案的权限模型设计。
> 完整调研报告见 [调研-社区项目深度分析.md](./调研-社区项目深度分析.md)。

### 4.1 行业权限模型对比

| 工具 | 角色数量 | 角色列表 | 定位 |
|------|---------|---------|------|
| Kitsu (CGWire) | 6 | Studio Manager / Production Manager / Supervisor / Artist / Vendor / Client | 中大型动画/VFX 工作室，多部门协作 |
| ShotGrid (Autodesk) | 4+ | Admin / Manager / Artist / Vendor + 自定义角色 | 行业标准，支持字段级权限配置 |
| Figma | 4 | Owner / Admin / Editor / Viewer | SaaS 协作工具标杆，权限继承模型 |
| Seedance 2.0 | 2 | admin / user | 轻量级 AI 视频生成平台 |

### 4.2 各工具权限细节

**Kitsu 6 级角色的具体职责：**

- **Studio Manager**：全系统访问，定义任务类型/状态/资产类型，管理所有项目和用户
- **Production Manager**：项目级全权控制，可批量创建资产/镜头（支持 CSV 导入），编辑任何评论
- **Supervisor**：部门级读写权限，只能在自己部门内分配任务，可查看客户评论
- **Artist**：仅能操作自己被分配的任务（上传媒体、修改状态、评论），登录首页为 "My Tasks"
- **Vendor**：比 Artist 更严格，只能看到被明确分配的任务，未分配的工作完全不可见
- **Client**：只读访问，只能查看所属项目的资产/镜头/统计，不能看到任务分配

**Figma 权限继承机制：**

```
Organization → Workspace → Team → Project → File
         权限默认向下继承，可在任意层级覆盖
```

- 每个团队/项目/文件有唯一 Owner
- Editor 继承自 Team 级别的编辑权限，Project 级别可覆盖为 Viewer
- 这种"默认继承 + 按需覆盖"模式被广泛认为是 SaaS 权限设计的最佳实践

**RBAC 最佳实践（来自 Oso、EnterpriseReady 等行业指南）：**

1. **最小权限原则**：用户只授予完成任务所需的最低权限
2. **避免角色爆炸**：不要为微小权限差异创建新角色；过多角色会使系统变得和管理个体权限一样复杂
3. **初期简单，按需扩展**：从 2-3 个角色开始，验证需求后再增加粒度
4. **租户隔离**：SaaS 环境中必须严格隔离不同团队/组织的数据

### 4.3 为什么 admin/user 二元模型是合理的

经过调研对比，我们的二元角色模型是正确的简化，理由如下：

1. **场景差异**：Kitsu 的 6 级角色服务于多部门协作流程（建模→绑定→动画→合成→灯光），每个部门有独立的 Supervisor。Seedance 2.0 的所有用户做的是同一件事（AI 视频生成），不存在部门划分需求。

2. **Vendor/Client 不适用**：Kitsu/ShotGrid 的 Vendor 角色面向外包团队，Client 角色面向甲方审片。我们的平台是自用工具，不涉及外部协作方。

3. **RBAC 最佳实践验证**：行业共识是"初期简单，按需扩展"，避免角色爆炸。admin/user 覆盖了当前全部需求：admin 管理即梦账号和系统配置，user 管理自己的项目和任务。

4. **Figma 的启示**：即使是 Figma 这样的大规模协作工具，核心也只有 4 级角色（Owner/Admin/Editor/Viewer）。我们的 admin 对应 Admin+Owner，user 对应 Editor。当用户规模小于 50 时，2 级完全足够。

### 4.4 未来扩展路径

如果第二阶段团队机制上线后需要更细粒度的权限，建议的扩展路径：

**阶段一（当前）：** 系统级 admin/user

**阶段二（团队机制 M6）：** 在 `team_members.role` 上增加团队级角色

```
系统角色：admin / user（不变）
团队角色：owner / member（新增，仅在 team_members 表中）
```

- team owner：管理团队设置、邀请成员、管理团队内项目
- team member：在团队内创建/管理自己的项目和任务

**阶段三（如果需要）：** 参考 Figma 模式增加 Viewer 角色

```
系统角色：admin / user
团队角色：owner / editor / viewer
```

- viewer：只能查看团队项目和任务，不能创建或修改（适用于客户预览场景）

这个扩展路径与 `team_members` 表已预留的 `role` 字段完全兼容，不需要额外的数据库变更。
