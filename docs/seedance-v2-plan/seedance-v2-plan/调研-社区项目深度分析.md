# 社区项目深度分析 — 影视制作管理工具调研报告

> 调研日期：2026-04-16
> 目的：为 Seedance 2.0 功能升级（团队协作影视制作平台）提供可借鉴的设计模式

## 一、Kitsu（CGWire 开源项目）

### 1.1 项目简介

Kitsu 是 CGWire 开发的开源影视制作协作平台，面向动画和 VFX 工作室。提供 GPL v3 自托管版和商业云版。后端 API 称为 Zou（Python + Flask + PostgreSQL），前端为 Vue.js SPA。每日超过 10 万次 API 调用，被众多工作室采用。

- GitHub: https://github.com/cgwire/kitsu
- 文档: https://kitsu.cg-wire.com/
- API 文档: https://zou.cg-wire.com/

### 1.2 数据模型

Kitsu 的核心实体及关系：

```
Production (项目)
├── Episode (集，TV Show 类型时启用)
│   └── Sequence (序列)
│       └── Shot (镜头)
├── Sequence (短片类型，直接挂在项目下)
│   └── Shot
├── AssetType (资产类型)
│   └── Asset (资产)
├── Edit (剪辑)
└── Concept (概念设计)

每个实体都可以挂载：
├── Task (任务，关联 TaskType 和 TaskStatus)
│   ├── Comment (评论)
│   └── PreviewFile (预览文件/版本)
└── Metadata (自定义元数据字段)
```

**关键实体属性：**

| 实体 | 核心字段 | 说明 |
|------|----------|------|
| Production | name, type (short/tvshow/featurefilm), status | 项目容器，type 决定是否启用 Episode |
| Episode | name, production_id | 仅 TV Show 类型启用 |
| Sequence | name, episode_id?, production_id | 序列，介于集和镜头之间 |
| Shot | name, sequence_id, nb_frames, frame_in, frame_out | 镜头，包含帧范围 |
| Task | task_type_id, task_status_id, entity_id, assignees | 任务，关联到任何实体 |
| PreviewFile | task_id, revision, extension, status | 版本/预览，revision 自增 |

**与 Seedance 2.0 对比：**
- Kitsu 有 Sequence 层级（我们的方案省略了，直接 Episode → Shot）
- Kitsu 的 Task 是独立实体可挂载到任何对象，我们的 Task 是执行层概念
- Kitsu 的 PreviewFile 等同于我们的"版本"概念

### 1.3 权限模型

Kitsu 实现了 6 级角色体系：

| 角色 | 范围 | 核心权限 |
|------|------|----------|
| **Studio Manager** | 全系统 | 等同超级管理员，管理所有项目、用户、全局设置、任务类型定义 |
| **Production Manager** | 项目级 | 继承 Supervisor 权限 + 创建资产/镜头（支持 CSV 批量导入）、编辑任何评论、管理播放列表 |
| **Supervisor** | 部门级 | 继承 Artist 权限 + 分配部门内任务、查看客户评论、管理部门播放列表和工时表 |
| **Artist** | 个人任务 | 仅能操作自己被分配的任务：上传媒体、修改状态、评论自己的工作 |
| **Vendor** | 极度受限 | 类似 Artist 但只能看到被明确分配的任务，未分配的工作完全不可见 |
| **Client** | 只读 | 只能查看所属项目的资产/镜头/统计，不能看到任务分配和其他用户评论 |

**权限控制特点：**
- 用户必须被加入项目团队才能接收任务分配
- Artist 登录后首页是 "My Tasks"（个人任务视图）
- Supervisor 只能在自己部门内分配任务
- 角色按项目分配，同一用户在不同项目可有不同角色

### 1.4 文件命名规范

Kitsu/Zou 使用 JSON 配置的 File Tree Descriptor 定义文件路径和命名：

**文件夹路径模板：**
```
Shot: <Project>/shots/<Sequence>/<Shot>/<TaskType>
Asset: <Project>/assets/<AssetType>/<Asset>/<TaskType>
```

**文件名模板：**
```
Shot: <Project>_<Sequence>_<Shot>_<TaskType>
Asset: <Project>_<AssetType>_<Asset>_<TaskType>
```

- 支持 `uppercase`/`lowercase` 风格参数
- 区分 working（工作文件）和 output（输出文件）两个上下文
- 模板标签（如 `<Project>`、`<Shot>`）自动替换为实体名称

**实际示例：** `LGC_100-000.mov`（项目 LGC，序列 100，镜头 000）

### 1.5 API 设计模式

Zou API 遵循标准 RESTful 设计：

```
GET    /api/data/projects                    # 项目列表
GET    /api/data/projects/:id                # 项目详情
GET    /api/data/projects/:id/sequences      # 项目下的序列
GET    /api/data/projects/:id/episodes       # 项目下的集
GET    /api/data/sequences/:id/shots         # 序列下的镜头
POST   /api/data/projects/:id/shots          # 创建镜头
GET    /api/data/tasks/:id/comments          # 任务评论
POST   /api/data/tasks/:id/preview-files     # 上传预览文件
```

- 认证：JWT Token
- 通过 Nginx 代理转发到 Gunicorn（端口 5000 主 API，5001 WebSocket 事件）
- Python 客户端 Gazu 提供高级封装

### 1.6 UI 布局

- **项目列表页**：卡片式展示，显示项目封面、名称、状态
- **镜头列表页**：表格视图，列包含缩略图、镜头名、各任务类型的状态色块（类似看板）
- **任务详情面板**：右侧滑出面板，包含状态选择器、评论列表、预览文件历史
- **My Tasks 页**：Artist 首页，按任务类型和状态分组展示个人任务
- **播放列表**：视频序列预览，支持逐帧评审和标注

## 二、ShotGrid / Flow Production Tracking（Autodesk）

### 2.1 项目简介

ShotGrid（现已更名为 Flow Production Tracking）是 Autodesk 的商业影视制作管理平台，是行业事实标准。提供 Web UI + Pipeline Toolkit（文件系统自动化）+ API。

- 官网: https://www.autodesk.com/products/flow-production-tracking
- Toolkit GitHub: https://github.com/shotgunsoftware
- 开发者文档: https://developers.shotgridsoftware.com

### 2.2 数据模型

ShotGrid 使用高度可配置的实体系统：

```
Project
├── Sequence
│   └── Shot
│       └── Version (关联 Task)
├── Asset
│   └── Version
├── Task (关联到 Shot/Asset/Sequence 等任何实体)
│   └── Note (评论/反馈)
├── PublishedFile (发布文件)
└── CustomEntity01-30 (最多 30 个自定义实体)
```

**核心实体：**

| 实体 | 说明 |
|------|------|
| Project | 项目，包含所有相关数据 |
| Sequence | 序列，镜头的分组 |
| Shot | 镜头，核心制作单元 |
| Version | 版本，关联到 Task，包含媒体文件 |
| Task | 任务，可挂载到任何实体 |
| Note | 评论，支持 @提及和附件 |
| PublishedFile | 正式发布的文件 |

### 2.3 Toolkit 文件命名模板

ShotGrid Toolkit 通过 `templates.yml` 配置文件定义命名规范：

**模板键（Template Keys）：**
```yaml
keys:
  Sequence:
    type: str
  Shot:
    type: str
  Step:
    type: str          # 流程步骤（如 comp, anim）
  version:
    type: int
    format_spec: "03"  # 零填充3位
  name:
    type: str
    filter_by: alphanumeric
```

**路径模板示例：**
```yaml
paths:
  shot_root:     sequences/{Sequence}/{Shot}/{Step}
  shot_work:     @shot_root/work/{application}/{name}.v{version}.{extension}
  shot_publish:  @shot_root/publish/{application}/{name}.v{version}.{extension}
```

**文件名模板：**
```
{Shot}_{name}_{nuke.output}_v{version}.{iteration}
```

**关键特性：**
- 模板键自动匹配 Schema 中的动态文件夹
- 支持 `@alias` 引用避免重复
- 版本号使用数字格式（v001, v002...）
- `{application}` 键自动识别 DCC 软件

### 2.4 权限模型

ShotGrid 使用基于模板的权限角色系统：

| 角色 | 范围 | 权限 |
|------|------|------|
| **Admin** | 全站 | 创建/归档项目、Schema 变更、用户管理、身份切换 |
| **Manager** | 项目级 | 默认可见所有项目，管理指定项目的任务和资源 |
| **Artist** | 项目成员 | 查看/更新自己的任务、提交版本、阅读反馈 |
| **Vendor** | 极度受限 | 只能看到被分配的任务和资产，以及针对自己的 Note |

**权限配置特点：**
- 支持创建自定义角色（基于模板派生）
- 权限粒度到字段级别（某角色是否能编辑某实体的某字段）
- 支持页面级权限（限制某角色不能访问某些页面）
- Admin 和 Manager 的区别主要在于是否能修改全局设置

### 2.5 UI 布局

- **详情页**：类电子表格的列表视图，支持内联编辑、列排序和过滤
- **甘特图/时间线**：任务调度可视化
- **媒体审阅**：内嵌播放器，支持逐帧标注和 Drawing 工具
- **仪表板**：可定制的统计面板，支持拖拽组件
- **全局搜索**：跨实体搜索

## 三、AYON（原 OpenPype）

### 3.1 项目简介

AYON 是 Ynput 开发的开源 VFX/动画 pipeline 平台，由 OpenPype 演进而来。采用 Server + Addon 架构，提供一致的文件结构、命名规范和版本管理。LGPL 开源。

- 官网: https://ynput.io/ayon/
- GitHub: https://github.com/ynput
- 文档: https://docs.ayon.dev

### 3.2 核心概念

**Project Anatomy（项目解剖）：** AYON 的核心配置概念，定义项目的文件结构和命名规范。

```
Anatomy 配置
├── Roots（存储根路径）
│   ├── work: /mnt/projects/{project}/work
│   └── publish: /mnt/projects/{project}/publish
├── Templates（路径模板）
│   ├── work: {root[work]}/{folder[name]}/{task[name]}/{app}/{file}
│   └── publish: {root[publish]}/{folder[name]}/{task[name]}/{family}/{file}
└── Entity Naming（命名规则）
    └── 从 label 自动生成标准化 name
```

**模板键：**
- `{root[work]}` / `{root[publish]}`：存储根路径引用
- `{folder[name]}`：AYON 层级结构中的文件夹名
- `{task[name]}`：任务名称
- `{family}`：发布类型（render, model, etc.）
- `{version}`：版本号

### 3.3 文件命名与版本管理

- **Work Area**：工作场景文件，自动版本递增
- **Publish Area**：不可变的发布文件，版本化存储
- 发布后自动验证和质量检查
- 支持快速回滚到任意历史版本
- Entity Naming 设置控制如何从 label 自动生成标准化 name

### 3.4 层级结构

AYON 使用灵活的 Folder 概念替代固定层级：
```
Project
└── Folder (可嵌套任意层级)
    ├── 类型可以是: Shot, Sequence, Episode, Asset, etc.
    └── Task (任务)
        └── Version (版本)
```

**Entity Templates** 允许预定义层级模板，快速批量创建项目结构。

## 四、Prism Pipeline

### 4.1 项目简介

Prism 是面向 VFX/动画的艺术家友好型 pipeline 工具。Prism 2 核心代码基于 LGPL v3 开源。侧重于文件版本管理和 DCC 集成。

- 官网: https://prism-pipeline.com/
- GitHub: https://github.com/PrismPipeline/Prism

### 4.2 文件命名与版本管理

- 文件夹结构和文件名可通过模板自定义
- 预制子目录类型：assets, scenes, dailies
- 支持定义本地和网络共享项目文件夹
- 自动在保存时创建新版本，无需手动管理版本号
- 可按用户或创建日期浏览版本历史，支持查看用户评论

### 4.3 镜头管理

- 项目内创建 Assets 和 Shots
- Shots 关联到 Departments（部门）
- 支持创建 Tasks 并分配给部门
- 主要功能：文件夹结构创建、资产/镜头/任务组织、场景文件/几何缓存/渲染图像的版本化

## 五、AI 视频生成平台的团队功能

### 5.1 Runway ML

**定位：** 最成熟的 AI 视频生成平台之一，面向专业创作者和企业。

**团队功能：**
- Enterprise 计划提供团队协作、集中计费和用量分析
- 支持团队 workspace，多用户共享项目
- Gen-3 Alpha 能力、4K 导出和协作工具锁定在付费计划中
- 适合需要管理分布式营销团队的企业

**权限模型：** 分为 Individual / Team / Enterprise 三级，Enterprise 提供管理员仪表板和使用量报告。

### 5.2 Pika Labs

**定位：** 面向个人创作者和小团队的 AI 视频生成工具。

**协作方式：**
- 依托 Discord 社区进行实时协作
- 用户在 Discord 频道中直接生成视频、分享和获取反馈
- 提供 API 用于自定义集成
- 缺乏企业级团队管理功能

### 5.3 Kling AI

**定位：** 快手旗下的 AI 视频生成平台（Kling 3.0）。

**团队功能现状（2026）：**
- 基础的多用户访问和共享积分池
- 管理员可进行使用量报告和监督
- **不支持** 多用户同时编辑共享项目
- **缺少** 版本控制、审批流程等专业功能
- 部分第三方平台（如 LTX Studio）提供更完善的团队工作流集成

### 5.4 小结

| 平台 | 团队功能成熟度 | 核心协作模式 |
|------|--------------|-------------|
| Runway ML | 高 | 正式团队 workspace + 管理仪表板 |
| Pika Labs | 低 | Discord 社区 + API |
| Kling AI | 低 | 共享积分池 + 基础多用户 |

## 六、SaaS 协作工具的权限模型参考

### 6.1 Figma 权限模型

Figma 的权限层级设计被广泛认为是 SaaS 产品的最佳实践之一：

```
Organization（组织）
└── Workspace（工作空间，Enterprise）
    └── Team（团队）
        └── Project（项目）
            └── File（文件）
```

**角色层级：**

| 角色 | 权限 |
|------|------|
| Owner | 团队唯一所有者，可转让，最高权限 |
| Admin | 管理团队设置、共享库、成员管理 |
| Editor | 创建/编辑文件、添加成员 |
| Viewer | 只读 + 评论 |

**关键设计原则：**
- **权限继承**：项目权限默认继承自团队权限
- **可覆盖**：可在项目/文件级别覆盖继承的权限
- **每个团队/项目/文件都有唯一 Owner**
- 简洁的四级角色覆盖绝大多数场景

### 6.2 RBAC 最佳实践（小型 SaaS 应用）

基于行业经验总结的小型 Web 应用 RBAC 实践：

1. **最小权限原则**：用户只授予完成任务所需的最低权限
2. **避免角色爆炸**：不要为微小权限差异创建新角色，过多角色会使 RBAC 变得和管理个体权限一样复杂
3. **租户隔离**：SaaS 环境中必须严格隔离不同团队/组织的数据
4. **审计日志**：记录权限变更、角色变更和访问尝试
5. **定期审查**：定期审计角色和权限分配
6. **初期简单，按需扩展**：从 2-3 个角色开始，验证需求后再增加粒度

## 七、文件命名行业标准

### 7.1 Netflix VFX 命名规范

Netflix 是行业中对文件命名要求最严格的客户之一：

**Shot 命名格式：**
```
{showID}_{episode}_{seq}_{scene}_{shotID}
示例: AGM_104_065_010
```

**Version 命名格式：**
```
{shot}_{task}_{vendorID}_{version}
示例: AGM_104_065_010_comp_NFX_v005
```

**规则：**
- showID: 2-3 字符项目缩写（必填）
- episode: 3 位数字，如 104 = 第 1 季第 4 集（剧集必填）
- shotID: 3-4 位数字，通常以 10 为步进（如 0010, 0020）
- version: 必须以 `v` 开头，3 位数字（v001, v002...）
- 各段之间用下划线 `_` 分隔

### 7.2 通用 VFX 文件命名规范

行业通用格式：
```
{Project}_{Sequence}_{Shot}_{Element}_{version}_{frameRange}.{ext}
示例: PROJ_SEQ001_SHOT001_BG01_v001_[1001-1058].exr
```

**最佳实践：**
- 只使用字母数字、下划线 `_`、连字符 `-`、句点 `.`
- 不使用空格
- 版本号统一格式（v001 或 V001）
- 可选添加部门/艺术家缩写（如 _JD 后缀）
- 大小写保持一致（推荐全小写或 CamelCase）

### 7.3 各工具的命名模式对比

| 工具 | 分隔符 | 版本格式 | 示例 |
|------|--------|---------|------|
| Kitsu/Zou | `_` | revision 自增 | `LGC_100_000_animation` |
| ShotGrid Toolkit | `_` + `.v` | v001 数字 | `SHOT001_comp_v001.exr` |
| Netflix | `_` | v001 数字 | `AGM_104_065_010_comp_NFX_v005` |
| AYON | 可配置 | 数字自增 | 由 Anatomy 模板决定 |
| Seedance 2.0（方案） | `-` | A-Z 字母 | `SN-1-1-C-ZS.mp4` |

## 八、对 Seedance 2.0 的借鉴建议

### 8.1 可直接采用的设计

| 领域 | 借鉴来源 | 具体建议 |
|------|---------|---------|
| **权限模型** | Figma + Kitsu | 保持 admin/user 二元角色是正确的。Figma 证明了简洁的角色层级（4 级）就能覆盖大多数场景。Kitsu 的 6 级角色对我们来说过度复杂。当前方案的 admin + user 在第一阶段完全足够。 |
| **权限继承** | Figma | 未来做团队机制时，采用 Figma 的权限继承模式：团队权限 → 项目权限 → 资源权限，默认继承，可覆盖。 |
| **文件命名中的版本号** | Netflix + ShotGrid | 当前方案使用字母版本号（A, B, C...）是一个不错的简化。行业标准用 v001 数字格式，但对于 AI 视频生成场景，字母版本更直观且短小。保持当前方案。 |
| **文件名分段结构** | Netflix + 通用规范 | 当前方案 `{项目代号}-{集数}-{镜头号}-{版本号}-{用户昵称}.mp4` 的结构合理。与 Netflix 的 `{showID}_{episode}_{shot}_{task}_{vendor}_{version}` 异曲同工，只是用连字符分隔且更简洁。 |
| **Artist 首页 = My Tasks** | Kitsu | 普通用户登录后应该看到自己的任务列表，而不是空白页面。当前方案中普通用户看到 "仅自己的" 项目/任务，方向正确。 |
| **版本在完成时分配** | ShotGrid + Kitsu | 当前方案中版本号在 output 任务完成时分配是正确的，与行业工具的做法一致（ShotGrid 的 Version 也是在 publish 时创建）。 |

### 8.2 需要简化的设计

| 领域 | 行业做法 | 简化建议 |
|------|---------|---------|
| **Sequence 层级** | Kitsu/ShotGrid 有 Sequence 层 | 当前方案省略 Sequence，直接 Episode → Shot，对于 AI 视频场景足够。Sequence 在传统 VFX 中用于镜头分组（如一个追车戏），但 AI 视频通常不需要这么细的组织粒度。保持当前三级结构（项目→集→镜头）。 |
| **Task 模型** | Kitsu 的 Task 是独立实体 | 我们的 draft/output 双层模型已经很好地服务于 AI 视频生成场景。不需要引入独立的 Task 实体。shot_id 挂在现有 tasks 表上是正确的融合方式。 |
| **File Tree 配置** | Kitsu/AYON 的 JSON/YAML 模板系统 | 对我们来说过度工程化。我们只有一种输出格式（.mp4 视频），不需要可配置的文件树。硬编码文件名拼接逻辑足够。 |
| **部门/TaskType 概念** | Kitsu 按部门（Layout, Animation, Compositing）组织 | AI 视频生成不涉及多部门协作流程，不需要 TaskType 概念。所有用户做的都是同一件事：生成视频。 |
| **自定义元数据** | ShotGrid 支持 30 个自定义实体 | 不需要。保持固定 schema，避免复杂性。 |
| **Pipeline Toolkit** | ShotGrid Toolkit 自动化文件系统操作 | 我们是 Web 应用，不涉及本地文件系统管理。文件命名只在下载/代理时拼接。 |

### 8.3 Seedance 2.0 独有的需求

以下需求在所有调研工具中都没有直接对应：

| 需求 | 说明 | 建议 |
|------|------|------|
| **即梦账号管理** | 核心资产，需要隔离普通用户对 SessionID 的访问 | 参考 SaaS 的 API Key 管理模式。当前方案（admin 可管理，user 只看摘要）是合理的。类似于 Runway 的 Enterprise 积分管理。 |
| **AI 视频生成轮询** | 前端每 3 秒轮询任务状态，与传统 pipeline 的手动提交/审核完全不同 | 这是实时性需求，传统工具用 WebSocket 事件推送（Kitsu 的 Zou 用端口 5001 推送事件）。未来可考虑 WebSocket 替代轮询，但当前轮询方案在用户规模较小时足够。 |
| **一个 Draft 展开多个 Output** | AI 视频生成特有的"一次生成多个候选"模式 | 这在传统 pipeline 中没有对应概念。ShotGrid 的 Version 是 1:1 的（一个 publish 对应一个 version）。我们的 draft → 多 output 模式是合理的创新。 |
| **浏览器代理/反爬** | Playwright 无头浏览器绕过平台反爬 | 完全是 Seedance 特有的技术实现，无行业参考。 |
| **积分/配额管理** | 即梦平台有使用配额限制 | 类似 Runway 的积分系统和 Kling 的共享积分池。可参考 Runway Enterprise 的团队配额管理思路。 |

### 8.4 总结建议

**当前方案评估：优秀。** 经过与行业工具的对比，Seedance 2.0 的升级方案在以下方面做出了正确的取舍：

1. **层级结构**：三级（项目→集→镜头）对 AI 视频场景足够，不需要 Sequence 层
2. **权限模型**：admin/user 二元角色简洁有效，避免了角色爆炸
3. **版本号**：字母版本号（A, B, C...）比数字版本号（v001）更适合轻量场景
4. **文件命名**：连字符分隔的扁平格式 `SN-1-1-C-ZS.mp4` 简洁直观
5. **Task 融合**：shot_id 挂载在现有 draft/output 模型上，避免了引入新的任务系统
6. **渐进增强**：镜头关联可选，不破坏现有体验

**唯一建议调整：**
- 考虑在文件命名中将连字符 `-` 改为下划线 `_`，以更符合行业惯例（Netflix、ShotGrid 均用下划线）。但连字符也可接受，且在 URL 中更友好。这是风格选择，不影响功能。
