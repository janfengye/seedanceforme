# 镜头管理与文件命名方案 — 四角色讨论纪要与最终方案

> 讨论日期：2026-04-16
> 参与角色：产品经理、前端程序员、后端程序员、架构师

---

## 一、讨论纪要

### 1. 产品经理视角

**核心观点：**

- 当前系统的"项目"是扁平的任务容器，用户无法按集/镜头组织内容。需要引入 **项目 → 集 → 镜头 → 版本** 四级结构，但不能强制所有用户都走这个流程。
- 参考行业工具（ShotGrid/Kitsu）的做法：Kitsu 中文件命名为 `production_sequence_shot`（如 `LGC_100-000.mov`），ShotGrid 的 Toolkit 通过配置定义文件夹结构和命名规范。我们采用 `{项目代号}-{集数}-{镜头号}-{版本号}-{用户昵称}.mp4` 的格式是合理的。
- **过渡策略**：现有项目不强制要求设置代号/集/镜头。只有用户主动创建集和镜头后，生成任务才能关联镜头并获得规范文件名。未关联镜头的任务保持现有体验。

**交互流程设计：**

1. 项目列表页增加「项目代号」列，点击项目进入详情页
2. 项目详情页展示集/镜头树形结构，支持内联创建/编辑/删除
3. 生成页面（单任务、批量）增加可选的镜头选择器：选项目 → 选集 → 选镜头
4. 镜头可预填 prompt 和参考图，用户选择镜头后自动填充

**争议点：**
- 产品经理主张镜头选择应为可选（不破坏现有"快速生成"体验），架构师同意。
- 产品经理希望版本号在生成页面实时预览（如"将生成版本 C"），后端程序员认为只能在提交时确定（并发安全），最终达成共识：**提交前显示预估版本号，实际版本号以后端分配为准**。

---

### 2. 前端程序员视角

**核心观点：**

- 当前 `SingleTaskPage.tsx` 是独立的生成页面，不关联任何项目/镜头。需要在其中增加可选的镜头选择级联组件。
- `BatchManagement.tsx` 已经有项目选择逻辑，可以在项目选择后增加镜头关联。
- 需要新建 `ProjectDetailPage.tsx` 作为项目详情页（集/镜头管理）。

**组件设计：**

```
ShotSelector (新组件)
├── ProjectSelect (下拉选择项目)
├── EpisodeSelect (下拉选择集)
└── ShotSelect (下拉选择镜头 + 新建镜头入口)
```

**关键交互：**
- 镜头选择器在生成页作为可折叠区域，默认收起
- 选中镜头后，自动填充镜头预设的 prompt（用户可覆盖）
- 批量生成中，每行 draft 可独立关联不同镜头
- 下载管理页新增「文件名」列

**争议点：**
- 前端程序员提出批量生成中每行关联镜头会增加 UI 复杂度。产品经理同意分两期：第一期只在单任务生成页支持镜头关联，第二期扩展到批量。最终架构师建议**第一期同时支持两者，批量只需在每行加一个镜头选择下拉即可，复杂度可控**。团队接受。

---

### 3. 后端程序员视角

**核心观点：**

- 数据库迁移必须向后兼容，所有新字段都有默认值，现有数据不受影响。
- 版本号计算是关键难点：SQLite 没有 `SELECT ... FOR UPDATE`，但 better-sqlite3 是同步的（单线程），天然避免并发问题。
- 文件命名拼接逻辑放在下载/代理接口中，不修改视频存储路径。

**版本号并发安全分析：**

```
better-sqlite3 特性：
- 所有操作是同步的，Node.js 单线程执行
- 不存在两个请求同时执行 SQL 的情况
- 因此 COUNT + INSERT 的版本号计算天然安全
- 无需额外加锁机制
```

**边界情况处理：**
- 用户昵称包含特殊字符 → 文件名中只保留字母数字和中文，其余替换为 `_`
- 项目代号为空 → 不生成规范文件名，降级为 `video_{taskId}_{timestamp}.mp4`
- 镜头被删除但任务仍存在 → `version_label` 已持久化到 tasks 表，文件名不受影响
- 版本号超过 Z（26个）→ 继续为 AA, AB, ... AZ, BA, ...（与 Excel 列名逻辑一致）

**争议点：**
- 后端程序员建议在 tasks 表直接存储完整的 `filename` 字段（冗余但查询方便）。架构师认为会导致数据不一致风险（改昵称后旧文件名不更新）。最终共识：**tasks 表只存 `version_label` 和 `shot_id`，文件名在查询时动态拼接**。对于已下载的历史文件，文件名已固定不变。

---

### 4. 架构师视角

**核心观点：**

- 四级结构（项目→集→镜头→版本）与现有双层任务模型（draft→output）是**正交关系**，不需要互相替代。
- draft/output 是**任务执行层**的概念（一个 draft 展开为多个 output），而 项目/集/镜头 是**内容组织层**的概念。
- 融合方式：`shot_id` 挂在 draft 任务上，draft 展开的 output 继承 `shot_id`。版本号在 output 任务成功完成时分配。

**融合方案：**

```
内容组织层：Project → Episode → Shot
                                    ↓ (shot_id)
任务执行层：              Draft Task → Output Task 1
                                     → Output Task 2
                                     → Output Task 3

版本号分配时机：Output Task 状态变为 done 时
版本号归属：按 shot_id + user_id 维度递增
```

**关键设计决策：**

1. `shot_id` 加在 `tasks` 表上（而非单独的关联表），因为一个任务只能属于一个镜头
2. 版本号在 output 任务完成时分配（而非提交时），避免失败任务占用版本号
3. 批量任务（batches 表）不需要改动，batch 关联的是 task_ids，task 自身已有 shot_id
4. 未来扩展：如果需要"场景"（Scene）层级，可以在 Episode 和 Shot 之间插入，不影响现有结构

**争议点：**
- 架构师提出是否需要 `scenes` 表（参考 ShotGrid 的 Sequence 概念）。产品经理认为当前用户规模不需要，保持三级（项目→集→镜头）即可。最终共识：**第一版不加 scenes，预留扩展空间**。

## 一b、权限模型补充（2026-04-16 后追加）

> **核心原则：管理员管理结构，普通用户使用结构。**

经讨论确定，项目/集/镜头的增删改操作**仅限管理员**。设计理由：
- 项目结构是团队共享资源，应由管理员统一规划
- 与传统影视制作流程一致：制片人（管理员）规划镜头列表，艺术家（普通用户）在指定镜头下提交作品
- 参考 Kitsu 的 Production Manager / Artist 角色分工

| 操作类别 | 管理员 | 普通用户 |
|----------|--------|---------|
| 创建/编辑/删除项目 | ✅ | ❌ |
| 设置项目代号（code） | ✅ | ❌ |
| 创建/编辑/删除集 | ✅ | ❌ |
| 创建/编辑/删除镜头 | ✅ | ❌ |
| 查看项目/集/镜头 | ✅ | ✅ |
| 生成视频并关联镜头 | ✅ | ✅ |
| 查看镜头版本列表 | ✅ | ✅ |

**前端体现**：
- 项目详情页的 [+ 添加新集]、[+ 添加镜头]、[编辑]、[删除] 按钮：普通用户隐藏
- ShotSelector 组件中的 [+ 新建镜头] 入口：普通用户隐藏
- 批量管理页的"创建项目"按钮：普通用户隐藏

**后端体现**：
- 所有写操作 API（POST/PUT/DELETE）添加 `requireAdmin` 中间件
- GET 接口保持 `authenticate` 即可

```sql
-- 注意：SQLite 的 ALTER TABLE ADD COLUMN 添加的 REFERENCES 不会被实际检查，

  ON tasks(shot_id, user_id)
```

### 2.3 版本号计算逻辑

```javascript
// server/services/versionService.js

/**
 * 数字转版本标签（Excel 列名风格）
 * 0→A, 1→B, ..., 25→Z, 26→AA, 27→AB, ...
 */
export function numberToVersionLabel(n) {
  let label = '';
  let num = n;
  do {
    label = String.fromCharCode(65 + (num % 26)) + label;
    num = Math.floor(num / 26) - 1;
  } while (num >= 0);
  return label;
}

/**
 * 为已完成的 output 任务分配版本号
 * 调用时机：output 任务状态变为 done 时
 * 并发安全：better-sqlite3 同步执行，Node.js 单线程，天然安全
 * 注意：此假设依赖单进程部署。如未来改为多进程（如 PM2 cluster），
 *       需改用 BEGIN IMMEDIATE 事务或其他锁机制。
 */
export function assignVersionLabel(db, taskId) {
  const task = db.prepare('SELECT id, shot_id, user_id FROM tasks WHERE id = ?').get(taskId);
  if (!task || !task.shot_id) return null;

  const count = db.prepare(
    'SELECT COUNT(*) as cnt FROM tasks WHERE shot_id = ? AND user_id = ? AND version_label IS NOT NULL'
  ).get(task.shot_id, task.user_id).cnt;

  const label = numberToVersionLabel(count);

  db.prepare('UPDATE tasks SET version_label = ? WHERE id = ?').run(label, taskId);
  return label;
}
```

### 2.4 文件命名逻辑

```javascript
// server/services/filenameService.js

/**
 * 清理文件名中的特殊字符
 * 只保留字母、数字、中文、下划线、连字符
 */
function sanitizeForFilename(str) {
  if (!str) return '';
  return str.replace(/[^\w\u4e00-\u9fff-]/g, '_').replace(/_+/g, '_').replace(/^_|_$/g, '');
}

/**
 * 构建规范文件名
 * 返回 null 表示无法构建（缺少必要信息），应降级为默认文件名
 */
export function buildStandardFilename(task) {
  const { project_code, episode_number, shot_number, version_label, nickname } = task;

  // 所有字段都必须存在才能构建规范文件名
  if (!project_code || episode_number == null || shot_number == null || !version_label || !nickname) {
    return null;
  }

  const safeName = sanitizeForFilename(nickname);
  return `${project_code}-${episode_number}-${shot_number}-${version_label}-${safeName}.mp4`;
}

/**
 * 构建降级文件名（未关联镜头时使用）
 */
export function buildFallbackFilename(taskId) {
  const timestamp = Date.now();
  return `video_${taskId}_${timestamp}.mp4`;
}

/**
 * 获取任务的完整文件名信息（需关联查询）
 * SQL 示例：
 */
export const FILENAME_QUERY = `
  SELECT
    t.id,
    t.version_label,
    p.code AS project_code,
    e.episode_number,
    s.shot_number,
    u.nickname,
    u.nickname,
    u.email AS user_email
  FROM tasks t
  LEFT JOIN shots s ON t.shot_id = s.id
  LEFT JOIN episodes e ON s.episode_id = e.id
  LEFT JOIN projects p ON e.project_id = p.id
  LEFT JOIN users u ON t.user_id = u.id
  WHERE t.id = ?
`;
```

**注意**：上述查询中 `nickname` 字段依赖 M1（用户昵称）模块完成后才能使用。在 M1 完成前，可以用 email 的 @ 前部分作为临时昵称。

```
                        ▼
                    ┌─────────┐
                    │  Draft   │  task_kind='draft', shot_id=?, prompt=...
                    │  Task    │  （用户提交时设置 shot_id）
                    └────┬────┘
                         │ 展开（video_count=3）
                    ┌────┴────────────────┐
                    │         │           │
               ┌────┴───┐ ┌──┴─────┐ ┌──┴─────┐
               │Output 1│ │Output 2│ │Output 3│
               │idx=0   │ │idx=1   │ │idx=2   │
               └────┬───┘ └──┬─────┘ └──┬─────┘
                    │        │          │
              status=done  status=done  status=error
              version=C    version=D    (无版本号)
```

**关键规则：**

1. `shot_id` 存储在 draft 和 output 任务上（output 从 draft 继承）
2. 版本号仅在 output 任务 `status` 变为 `done` 时分配
3. 失败的 output 任务不占用版本号
4. 同一个 draft 的多个 output 会依次获得不同版本号（C, D, E...）
5. 取消的任务不占用版本号

**代码修改点：**
- `videoGenerator.js` 中任务完成回调处调用 `assignVersionLabel()`
- `taskService.js` 中创建 output 任务时从 draft 继承 `shot_id`
- 下载接口中通过关联查询拼接文件名

### 2.7 文件命名边界情况

| 边界情况 | 处理方式 |
|----------|----------|
| 未关联镜头 | 降级为 `video_{taskId}_{timestamp}.mp4` |
| 项目代号为空 | 同上，降级处理 |
| 用户无昵称 | 使用 email 的 `@` 前部分，经 sanitize 处理 |
| 昵称含特殊字符（空格、emoji等） | `sanitizeForFilename()` 替换为 `_` |
| 昵称为纯特殊字符 | sanitize 后为空，降级为 `user_{userId}` |
| 版本号超过 Z | 继续为 AA, AB, ..., AZ, BA, ... |
| 镜头被删除 | 已分配的 `version_label` 保留在 tasks 表，但新下载时因 shot 不存在将降级文件名 |
| 集/镜头编号有间隔（如删除后重建） | 编号由用户或系统自增指定，间隔不影响文件名 |
| 同一镜头多用户同时生成 | 各用户版本号独立递增，互不影响 |
| 重命名项目代号 | 新生成任务用新代号，已有任务的 `version_label` 不变，文件名动态拼接会自动更新 |

### 2.9 风险与注意事项

| 风险 | 说明 | 应对 |
|------|------|------|
| 迁移脚本兼容性 | SQLite 的 ALTER TABLE 功能有限，不支持一次添加多列 | 每个 ALTER TABLE 单独一条语句 |
| `shouldSkipMigration` 逻辑 | 已有三个迁移被跳过，新迁移需测试全新数据库和升级场景 | 新迁移使用 `IF NOT EXISTS`，在 schema.sql 中同步更新 |
| 用户昵称依赖 | M4 依赖 M1 的昵称字段，M1 未完成时文件名缺少昵称 | 临时使用 email 前缀作为 fallback |
| 大量镜头的性能 | 项目详情页一次加载所有集和镜头 | 前端分页或虚拟滚动（当集/镜头超过 100 时）；初期不做，观察实际使用量 |
| 删除集的级联影响 | 删除集会级联删除镜头，镜头关联的任务 shot_id 需处理 | 删除镜头时 tasks.shot_id 置 NULL（而非级联删除任务），需在 schema 中用 `ON DELETE SET NULL` |
| Docker 重建 | schema 变更需要重新构建 Docker 镜像 | 迁移脚本机制已有，正常流程即可 |

**重要修正**：shots 表的外键级联策略需调整。tasks 表的 shot_id 外键应使用 `ON DELETE SET NULL` 而非 `ON DELETE CASCADE`，确保删除镜头时不会误删任务：

```sql
-- tasks 表的 shot_id 外键（SQLite 不支持 ALTER 外键，通过应用层保证）
-- 删除镜头时，应用层先执行：
-- UPDATE tasks SET shot_id = NULL WHERE shot_id = ?
-- 然后再 DELETE FROM shots WHERE id = ?
```

## 三、行业参考

> 基于对 Kitsu、ShotGrid、AYON、Netflix VFX 规范等的深度调研。
> 完整调研报告见 [调研-社区项目深度分析.md](./调研-社区项目深度分析.md)。

### 3.1 各工具数据模型对比

| 维度 | Kitsu (CGWire) | ShotGrid (Autodesk) | AYON (Ynput) | Seedance 2.0 |
|------|---------------|---------------------|-------------|-------------|
| 层级结构 | Project → Episode → Sequence → Shot | Project → Sequence → Shot | Project → Folder（可嵌套任意层级） | Project → Episode → Shot |
| 是否有 Sequence 层 | 有（介于 Episode 和 Shot 之间） | 有（镜头分组） | 可配置（Folder 类型可以是 Sequence） | 无（简化省略） |
| 版本实体 | PreviewFile（挂在 Task 上，revision 自增） | Version（独立实体，关联 Task） | Version（关联到 Task） | version_label 字段（存在 tasks 表上） |
| 任务模型 | Task 独立实体，可挂载到任何对象 | Task 独立实体，可挂载到任何对象 | Task 挂在 Folder 上 | draft/output 双层模型，shot_id 关联 |
| 自定义实体 | 支持自定义 Metadata 字段 | 最多 30 个 CustomEntity | Folder 类型可自定义 | 固定 Schema |

**Kitsu 核心实体关系：**

```
Production (type: short/tvshow/featurefilm)
├── Episode (仅 tvshow 类型)
│   └── Sequence
│       └── Shot (name, nb_frames, frame_in, frame_out)
├── AssetType → Asset
├── Edit (剪辑)
└── Concept (概念设计)

每个 Shot/Asset 可挂载:
├── Task (task_type_id, task_status_id, assignees[])
│   ├── Comment
│   └── PreviewFile (revision 自增, extension, status)
└── Metadata (自定义键值对)
```

**ShotGrid 核心实体：**

```
Project
├── Sequence → Shot → Version (media_file, task, user)
├── Asset → Version
├── Task (entity_id 多态关联到 Shot/Asset/Sequence)
│   └── Note (可 @提及, 支持附件)
└── PublishedFile (正式发布文件)
```

**省略 Sequence 的合理性：** Sequence 在传统 VFX 中用于将镜头按场景分组（如"追车戏"包含 20 个镜头）。AI 视频生成场景中，每个镜头通常是独立的创意单元，不需要场景级分组。直接 Episode → Shot 的简化是合理的。如果未来有需求，可在 Episode 和 Shot 之间插入 Scene 层级（方案 2.9 已提及）。

### 3.2 文件命名规范对比

#### Netflix VFX 命名规范（行业最严格标准之一）

**Shot 命名格式：**
```
{showID}_{episode}_{seq}_{scene}_{shotID}
示例: AGM_104_065_010
```

- showID: 2-3 字符项目缩写（必填）
- episode: 3 位数字，如 104 = 第 1 季第 4 集
- shotID: 3-4 位数字，以 10 为步进（0010, 0020），方便中间插入

**Version 命名格式：**
```
{shot}_{task}_{vendorID}_{version}
示例: AGM_104_065_010_comp_NFX_v005
```

- version: 必须以 `v` 开头 + 3 位数字（v001, v002...）

#### Kitsu/Zou File Tree Descriptor

Kitsu 通过 JSON 配置定义文件路径和命名模板：

```json
{
  "working": {
    "mountpoint": "/working_files",
    "root": "productions",
    "folder_path": {
      "shot": "<Project>/shots/<Sequence>/<Shot>/<TaskType>",
      "asset": "<Project>/assets/<AssetType>/<Asset>/<TaskType>"
    },
    "file_name": {
      "shot": "<Project>_<Sequence>_<Shot>_<TaskType>",
      "asset": "<Project>_<AssetType>_<Asset>_<TaskType>"
    }
  },
  "output": { ... }
}
```

- 标签（如 `<Project>`、`<Shot>`）自动替换为实体名称
- 支持 `uppercase`/`lowercase` 风格参数
- 区分 working（工作中）和 output（输出）两个上下文

#### ShotGrid Toolkit templates.yml

ShotGrid 通过 YAML 配置文件定义命名模板：

```yaml
keys:
  Sequence:  { type: str }
  Shot:      { type: str }
  Step:      { type: str }           # 流程步骤（comp, anim）
  version:   { type: int, format_spec: "03" }  # 零填充 3 位
  name:      { type: str, filter_by: alphanumeric }

paths:
  shot_root:    sequences/{Sequence}/{Shot}/{Step}
  shot_work:    @shot_root/work/{application}/{name}.v{version}.{extension}
  shot_publish: @shot_root/publish/{application}/{name}.v{version}.{extension}

# 文件名模板
strings:
  shot_version_name: "{Shot}_{name}_{nuke.output}_v{version}.{iteration}"
```

- `@alias` 引用避免路径重复
- 模板键自动匹配 Schema 中的动态文件夹
- 支持 `format_spec` 控制数字格式

#### AYON Project Anatomy

AYON 使用 Roots + Templates 配置体系：

```
Roots:
  work:    /mnt/projects/{project}/work
  publish: /mnt/projects/{project}/publish

Templates:
  work:    {root[work]}/{folder[name]}/{task[name]}/{app}/{file}
  publish: {root[publish]}/{folder[name]}/{task[name]}/{family}/{file}
```

- `{root[work]}` 引用存储根路径
- `{folder[name]}` 引用 AYON 层级结构
- Entity Naming 设置控制从 label 自动生成标准化 name
- 不建议在生产过程中修改 Anatomy（会导致路径不一致）

### 3.3 命名格式综合对比

| 工具/标准 | 分隔符 | 版本格式 | 完整示例 |
|-----------|--------|---------|---------|
| Netflix VFX | `_` | v001 (数字) | `AGM_104_065_010_comp_NFX_v005` |
| Kitsu/Zou | `_` | revision 自增 | `LGC_100_000_animation` |
| ShotGrid Toolkit | `_` + `.v` | v001 (数字) | `SHOT001_comp_v001.1.exr` |
| AYON | 可配置 | 数字自增 | 由 Anatomy 模板决定 |
| 通用 VFX 规范 | `_` | v001 (数字) | `PROJ_SEQ001_SHOT001_BG01_v001.exr` |
| **Seedance 2.0** | **`-`** | **A-Z (字母)** | **`SN-1-1-C-ZS.mp4`** |

**通用行业规则：**
- 只使用字母数字、下划线 `_`、连字符 `-`、句点 `.`
- 不使用空格
- 可选添加部门/艺术家缩写后缀
- 版本号格式统一且可排序

### 3.4 对 Seedance 2.0 命名方案的验证

我们的 `{项目代号}-{集数}-{镜头号}-{版本号}-{用户昵称}.mp4` 格式与行业标准对比：

| 对比维度 | 行业标准 | Seedance 2.0 | 评估 |
|----------|---------|-------------|------|
| 分隔符 | 下划线 `_` 为主 | 连字符 `-` | 可接受。连字符在 URL 中更友好，且视觉上更紧凑。风格选择，不影响功能。 |
| 版本号 | 数字 v001 | 字母 A-Z-AA | 合理简化。AI 视频每个镜头的版本数远少于传统 VFX（通常 <26 个），字母更直观紧凑。 |
| 用户标识 | vendorID 或省略 | 用户昵称 | 合理创新。传统 VFX 用 vendor 缩写，我们用昵称更适合团队内部场景。 |
| 帧范围 | 常包含 `[1001-1058]` | 不需要 | 正确。我们输出的是完整视频文件，不是图片序列。 |
| 任务类型 | 包含 comp/anim 等 | 不需要 | 正确。所有任务都是同一类型（AI 视频生成）。 |

**结论：** 当前命名方案是行业标准在 AI 视频生成场景下的合理简化，保留了核心信息（项目、集、镜头、版本、作者），去除了不适用的字段（帧范围、任务类型、vendor）。

## 四、总结

本方案在现有 draft/output 双层任务模型基础上，通过正交的内容组织层（项目→集→镜头）实现了四级结构管理。核心设计原则：

1. **向后兼容**：所有新功能可选，不破坏现有使用流程
2. **渐进增强**：用户可以逐步从"快速生成"过渡到"结构化管理"
3. **并发安全**：利用 better-sqlite3 同步特性，版本号计算天然安全
4. **动态拼接**：文件名在查询时动态生成，避免数据不一致
5. **优雅降级**：缺少任何必要字段时自动降级为默认文件名
