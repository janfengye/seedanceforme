#### S2. tasks 表 shot_id 外键的 ON DELETE 策略缺失/矛盾

**问题**：M3 文档（03-项目结构改造.md）中 tasks 表的 shot_id 外键没有指定 ON DELETE 策略：

```sql
ALTER TABLE tasks ADD COLUMN shot_id INTEGER DEFAULT NULL REFERENCES shots(id);
```

而讨论-镜头管理与文件命名方案.md 2.9 节明确指出应使用 `ON DELETE SET NULL`，且解释了 SQLite 的 ALTER TABLE 不支持修改外键约束，需要应用层保证。

**但更关键的是**：SQLite 的 `ALTER TABLE ADD COLUMN` 虽然语法上支持 `REFERENCES`，但实际上**不会强制执行外键约束**（SQLite 文档明确说明 ALTER TABLE 添加的列的外键约束不会被检查）。因此写不写 `REFERENCES shots(id)` 在功能上没有区别。

**修正建议**：
1. 在迁移脚本中去掉 `REFERENCES shots(id)`（避免误导开发者以为有外键保护）
2. 在应用层（删除 shot 的 API 中）添加：`UPDATE tasks SET shot_id = NULL WHERE shot_id = ?`，然后再 `DELETE FROM shots WHERE id = ?`
3. 在 M3 和 M4 文档中明确标注此限制

#### S3. 邀请码随机性不足

**问题**：M0 文档中邀请码生成逻辑为：

```javascript
crypto.randomBytes(5).toString('base64url').substring(0,8).toUpperCase()
```

`crypto.randomBytes(5)` 产生 5 字节 = 40 位熵，但 `base64url` 编码后取前 8 字符再 `toUpperCase()` 会丢失大小写区分，实际有效字符集为 `[A-Z0-9_-]`（约 38 个字符），8 位约 44 位熵。对于邀请码场景这是足够的，但**与格式描述矛盾**。

文档描述格式为 `/^[A-Z0-9]{8}$/`，但 `base64url` 编码会包含 `-` 和 `_` 字符，toUpperCase 后仍保留这些字符，不符合 `[A-Z0-9]` 的正则约束。

**修正建议**：使用明确的字符集生成：
```javascript
const CHARSET = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'; // 去掉易混淆的 0/O, 1/I
const bytes = crypto.randomBytes(8);
const code = Array.from(bytes).map(b => CHARSET[b % CHARSET.length]).join('');
```

#### S4. projects 表的 code 唯一索引范围不一致

**问题**：讨论-镜头管理与文件命名方案.md 中项目代号唯一索引定义为：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_projects_user_code
  ON projects(user_id, code) WHERE code IS NOT NULL;
```

这意味着唯一性按 `user_id` 隔离——不同用户可以使用相同的项目代号。

但 M6 中引入了团队机制，项目归属团队，团队内成员共享项目。如果两个用户在同一团队中使用相同的项目代号（如 `SN`），会导致文件名冲突（两个人的视频都叫 `SN-1-1-A-XXX.mp4`）。

**修正建议**：
- 第一阶段（所有人同一团队）：唯一索引改为全局唯一 `ON projects(code) WHERE code IS NOT NULL`
- 第二阶段（多团队）：改为 `ON projects(team_id, code) WHERE code IS NOT NULL`

### 中等问题（建议修复）

#### M1. 现有即梦账号 API 无 requireAdmin 中间件

**问题**：现有代码中 `/api/settings/session-accounts` 的 POST/PUT/DELETE 路由仅使用 `authenticate` 中间件，无 `requireAdmin`：

- `server/index.js:2148` POST — 仅 `authenticate`

- `server/index.js:2172` DELETE — 仅 `authenticate`

讨论-账号管理体系方案.md 已正确指出需要添加 `requireAdmin`，但现有代码的实际情况是这些 API 本身就按 `user_id` 隔离（每个用户管理自己的账号）。

**分析**：当前架构中即梦账号是按 `user_id` 绑定的（`jimeng_session_accounts.user_id`），M2 要将其改为"管理员专属管理"。这意味着需要决定：
1. 保持 per-user 模型但 UI 上只让管理员看到？
2. 还是改为全局模型（去掉 user_id 关联或改为 admin 的 user_id）？

**修正建议**：在 M2 文档中明确说明迁移策略：现有 per-user 账号如何处理。建议方案是保持 `user_id` 字段但实际只允许管理员操作，GET 接口对普通用户返回全局摘要。

#### M2. version_label 并发安全分析有一个隐含前提

**问题**：讨论-镜头管理与文件命名方案.md 中声称 better-sqlite3 同步特性保证了 COUNT + UPDATE 的版本号计算天然安全，这在 **单进程** 下是正确的。但如果将来部署多个 Node.js 进程（如 PM2 cluster 模式），这个假设就不成立了。

**修正建议**：在文档中添加注意事项："当前版本号并发安全依赖 Node.js 单进程 + better-sqlite3 同步执行。如果未来部署改为多进程，需要改用 SQLite 的 `BEGIN IMMEDIATE` 事务或其他锁机制。"

#### M3. FILENAME_QUERY 中缺少 nickname 字段

#### M4. 默认团队创建的 created_by 硬编码为 1

**问题**：讨论-账号管理体系方案.md 2.2 节中：

```sql
INSERT INTO teams (name, description, created_by) VALUES ('默认团队', '系统默认团队', 1);
```

硬编码 `created_by = 1`，假设管理员的 `users.id = 1`。在全新数据库（schema.sql 先创建表再插入管理员）中这是成立的，但如果有人手动删除并重建管理员，ID 可能不是 1。

**修正建议**：改为子查询：
```sql
INSERT INTO teams (name, description, created_by)
VALUES ('默认团队', '系统默认团队', (SELECT id FROM users WHERE role = 'admin' LIMIT 1));
```

#### M5. invitation_codes 表缺少 created_by 的外键级联策略

**问题**：M0 中 `invitation_codes.created_by` 定义为 `REFERENCES users(id)` 但未指定 ON DELETE 策略。如果创建邀请码的管理员被删除，会导致外键约束错误。

**修正建议**：添加 `ON DELETE CASCADE` 或 `ON DELETE SET NULL`（根据业务需求选择）。考虑到邀请码有审计作用，建议 `ON DELETE SET NULL` 并将字段改为 `created_by INTEGER DEFAULT NULL`。

---

#### M6. 注册接口改造缺少事务保证

**问题**：M0 邀请码注册流程需要：1) 校验邀请码 → 2) 创建用户 → 3) used_count += 1 → 4) 记录 invitation_usage。这四步需要在同一事务中完成，否则可能出现用户创建成功但邀请码使用次数未更新的情况。

**修正建议**：在文档中明确标注使用 better-sqlite3 的 `db.transaction()` 包装整个注册流程。

#### M7. GET /api/settings 和 PUT /api/settings 缺少认证

**问题**：现有代码中 `server/index.js:2116` 和 `server/index.js:2126` 的全局设置 API 没有 `authenticate` 中间件，任何人都可以读写全局设置。虽然不在本次升级范围内，但应在安全审查中标注。

**修正建议**：GET 添加 `authenticate`，PUT 添加 `authenticate, requireAdmin`。

## 五、优化建议

### 1. 合并迁移脚本

M3（episodes/shots）和 M4（version_label）的迁移可以合并为一个脚本 `20260417_add_shot_management.sql`，减少迁移文件数量。讨论-镜头管理与文件命名方案.md 已建议了这一点。

### 2. schema.sql 同步更新

每次添加迁移脚本后，应同步更新 `server/database/schema.sql`，确保全新数据库的 schema 与迁移后的数据库一致。同时在 `shouldSkipMigration` 中添加新迁移的跳过逻辑。

### 3. 新建服务文件拆分

当前 `server/index.js` 已超过 3000 行，建议新增的 API 路由按模块拆分为独立路由文件：
- `server/routes/profile.js` — 个人设置 API
- `server/routes/invitation.js` — 邀请码管理 API
- `server/routes/episodes.js` — 集管理 API
- `server/routes/shots.js` — 镜头管理 API
- `server/routes/admin-dashboard.js` — 管理统计 API

使用 Express Router 挂载到 `index.js`。

### 4. 数据迁移脚本执行顺序

建议的迁移脚本执行顺序（按日期前缀排序）：


### 5. 为 tasks 表 user_id 添加索引

```sql
CREATE INDEX IF NOT EXISTS idx_tasks_shot_user_version
  ON tasks(shot_id, user_id) WHERE version_label IS NOT NULL;
```

但 03-项目结构改造.md 和 04-版本号与文件命名.md 中没有提到此索引，应在最终迁移脚本中统一包含。
