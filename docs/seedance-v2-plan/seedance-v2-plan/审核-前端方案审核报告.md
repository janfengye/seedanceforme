**位置**：`src/types/index.ts`（现有代码）vs `00b-邀请码注册机制.md`
**问题**：现有 `RegisterCredentials` 接口定义为 `{ email, password, emailCode }`。M0 方案要求注册时增加 `invitation_code` 字段，但在 `设计-前端UI详细方案.md` 的类型定义变更章节（第 8 节）中完全没有提到 `RegisterCredentials` 的修改，也没有提到 `InvitationCode` 相关的前端类型定义。
**修正建议**：在 `设计-前端UI详细方案.md` 第 8 节增加：
```typescript
// RegisterCredentials 扩展
export interface RegisterCredentials {
  email: string;
  password: string;
  emailCode: string;
  invitation_code: string;  // 新增
}

// 邀请码相关类型（管理端）
export interface InvitationCode {
  id: number;
  code: string;
  note?: string;

  used_count: number;
  expires_at?: string;
  is_active: boolean;
  created_at: string;
  usedBy: { userId: number; email: string; usedAt: string }[];

```

## 五、产品体验优化建议

### 5.1 邀请码注册体验

**当前方案**：邀请码输入框放在注册表单最上方。
**建议**：
- 邀请码格式应提供实时校验（格式错误时即时提示，如"邀请码格式应为 8 位字母数字"），不必等到提交时才报错
- 无效邀请码的错误提示"邀请码无效或已过期"过于笼统，建议区分：已过期 / 已用完 / 不存在 / 已停用，给出不同提示
- 考虑支持邀请码 URL：`/register?code=SD-X7KM9A`，用户点击管理员分享的链接可自动填充邀请码

### 5.2 昵称未设置时的引导流程

**当前方案**：Sidebar 底部显示提示气泡，引导跳转 `/profile`。
**建议**：
- 缺少注册后首次登录的强制引导。建议在首次登录时弹出 Modal 引导设置昵称（不是阻断式，但足够显眼）
- 当用户在生成页尝试关联镜头且未设置昵称时，应直接弹出昵称设置 Modal（而非跳转到 `/profile` 页面再跳回来），减少上下文切换
- 文件名预览中昵称未设置时显示的 fallback（email 前缀）应在 ShotSelector 中用警告色标注

### 5.3 镜头选择器交互

**当前方案**：三级级联下拉，默认收起。
**优化建议**：
- 从项目详情页点击 [生成] 跳转到 `/?shot={id}` 时，ShotSelector 应自动展开并预选该镜头。URL 参数解析逻辑需在设计文档中明确
- 建议增加"最近使用"快捷选项，避免每次都从项目重新选择
- 选择镜头后预设 prompt 自动填充的交互建议增加一个过渡动画（如高亮闪烁），让用户明确感知到内容被填充了

### 5.4 管理员与普通用户的界面差异

**当前方案**：Settings 页按角色条件渲染；Admin 页仅管理员可见。
**优化建议**：
- 普通用户在 Settings 页看到的即梦账号摘要过于简单（只有数字），建议增加一个"系统状态"指示灯（绿/黄/红），直观表示当前是否可以提交任务
- 建议在生成按钮旁也显示一个小指示器，显示"当前可用账号: 3"，减少用户去 Settings 页查看的必要

### 5.5 新用户首次使用引导

**当前方案**：缺失。
**建议**：
- 新注册用户首次进入主页时，可以显示一个简洁的欢迎 Banner（可关闭），引导："1. 设置昵称 → 2. 创建项目 → 3. 开始生成"
- 空项目列表时使用 EmptyState 组件引导创建第一个项目

### 5.6 批量下载文件命名

**问题**：方案中只提到单个下载使用规范文件名，未明确批量下载（zip 包）的行为。
**建议**：批量下载时 zip 文件内的每个视频都应使用规范文件名。zip 文件本身命名建议为 `{项目代号}-batch-{日期}.zip`。
